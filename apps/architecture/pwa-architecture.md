# VeoLMS PWA Architecture

This document explains the Progressive Web App layer added to the VeoLMS frontend. The goal is to make the learning experience installable, resilient to temporary network loss, and safe to update without caching sensitive flows incorrectly.

---

## 1. Goals

The PWA feature is designed around four practical requirements:

- Let students install VeoLMS from supported desktop and mobile browsers.
- Keep the application shell available during short network interruptions.
- Show a clear offline fallback for pages that cannot be recovered from cache.
- Avoid caching sensitive authenticated, payment, admin, upload, and video operations.

This is intentionally a resilience layer, not a full offline LMS. Course purchase, video playback, admin actions, uploads, and authenticated mutations still require the network.

---

## 2. Implementation Entry Points

| Area | File | Responsibility |
|---|---|---|
| PWA plugin config | `apps/client/vite.config.ts` | Defines manifest, Workbox runtime caching, install assets, and navigation fallback rules |
| Update prompt | `apps/client/src/components/PWAUpdatePrompt.tsx` | Registers the service worker and asks the user before reloading into a new version |
| App mounting | `apps/client/src/main.tsx` | Mounts the update prompt globally beside the router and providers |
| Offline fallback | `apps/client/public/offline.html` | Static user-facing page for offline states that cannot be served by the SPA cache |
| Generated output | `apps/client/dist/sw.js`, `apps/client/dist/site.webmanifest` | Production build artifacts generated by `vite-plugin-pwa` |

---

## 3. Manifest Design

The manifest is generated as `site.webmanifest` with:

- `name` and `short_name`: `VeoLMS`
- `display`: `standalone`
- `start_url` and `scope`: `/`
- `orientation`: `portrait`
- `theme_color` and `background_color`: `#18181b`
- Education/productivity categories
- 192px and 512px icons with both `any` and `maskable` purposes
- App shortcuts for course browsing and the learner dashboard

The install assets are included directly through the PWA plugin so the app icon, favicon, logo, and offline page are available to the service worker build.

---

## 4. Service Worker Strategy

VeoLMS uses `vite-plugin-pwa` with Workbox. The service worker is generated during the production client build.

The registration mode is `prompt`, which means a new service worker does not force-refresh the learner in the middle of an active session. Instead, the frontend displays a small update prompt and reloads only when the user accepts.

Important settings:

- `cleanupOutdatedCaches: true` removes old cache versions.
- `skipWaiting: false` prevents a new worker from immediately taking over.
- `clientsClaim: false` avoids uncontrolled takeover of already-open tabs.
- `navigateFallback: /index.html` keeps normal SPA routes loadable from the app shell cache.
- `navigateFallbackDenylist` excludes `/api/`, `/admin`, and `/checkout` from generic SPA fallback handling.

---

## 5. Runtime Caching

The cache policy separates static shell assets from data that can change quickly or carry user-sensitive state.

| Cache | Strategy | Max Age | Purpose |
|---|---|---|---|
| `veolms-html` | NetworkFirst | 1 day | Page navigations prefer fresh HTML but can survive short outages |
| `veolms-js` | StaleWhileRevalidate | 7 days | Versioned JavaScript chunks load quickly and refresh in the background |
| `veolms-css` | StaleWhileRevalidate | 7 days | Versioned CSS loads quickly and refreshes in the background |
| `veolms-fonts` | CacheFirst | 365 days | Fonts are stable and safe to reuse |
| `veolms-images` | CacheFirst | 30 days | Public image assets and thumbnails stay fast |
| `veolms-manifest` | NetworkFirst | 1 day | Manifest updates propagate without long stale windows |
| `veolms-public-api` | NetworkFirst | 5 minutes | Public course and coupon validation GET responses remain briefly resilient |

Only public, non-sensitive GET API responses are eligible for runtime API caching. Requests with authorization headers, auth paths, payment paths, admin paths, and non-GET requests are excluded.

---

## 6. Sensitive Flow Boundaries

The PWA layer deliberately does not turn the following workflows into offline-first experiences:

- Authentication and refresh token flows
- Payment order creation and verification
- Admin dashboard, course editing, exports, imports, and uploads
- HLS video streaming and protected lesson access
- Any API mutation

This boundary keeps the service worker from returning stale or cached responses where correctness, access control, or payment state must come from the server.

---

## 7. Update Lifecycle

`PWAUpdatePrompt` registers the generated service worker with `virtual:pwa-register`.

When Workbox detects a waiting update:

1. `onNeedRefresh` sets `updateReady`.
2. The app displays a bottom prompt with `Later` and `Reload`.
3. `Later` dismisses the prompt for the current session.
4. `Reload` calls the registered updater with `reloadPage: true`.

This keeps updates visible without interrupting checkout, course editing, or an active learning session.

---

## 8. Offline Experience

The static offline page tells the user that saved pages may still open, but payments, videos, admin tools, and uploads require an internet connection.

The app shell may serve previously visited routes from cache, but the offline page is the fallback when the browser cannot recover a navigable page from cached HTML and assets.

---

## 9. Verification Checklist

- Build the client and confirm `dist/sw.js` and `dist/site.webmanifest` are generated.
- Open production over HTTPS and confirm the browser exposes an install option.
- Visit a public course page, disable network, and confirm cached shell behavior.
- Open checkout or admin while offline and confirm sensitive routes are not incorrectly served as successful API states.
- Deploy a changed build and confirm the update prompt appears instead of forcing an immediate refresh.
